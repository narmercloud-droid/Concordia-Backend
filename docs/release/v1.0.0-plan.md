# Release Plan — v1.0.0

Purpose: prepare, validate and publish Concordia Backend v1.0.0.

Phases

1. Prep
   - Finalize docs and changelog.
   - Ensure `npm run sdk:test` passes (CI-green).
   - Confirm `docs/openapi/openapi.yaml` is unchanged for this release.

2. Cut
   - Update versions in SDK manifests (`sdks/typescript/package.json`, `sdks/python/setup.py` / `pyproject.toml`) as needed.
   - Create local git tag `v1.0.0` (annotated). Do not push yet.

3. Publish SDKs
   - TypeScript: `npm publish` from `sdks/typescript` (after `npm ci` and build steps).
   - Python: build wheel/sdist and upload via `twine upload` to TestPyPI then PyPI.

4. Release backend
   - Create release branch `release/v1.0.0` if required.
   - Push tag and create GitHub Release from tag (draft first, attach changelog and notes).

5. Post-release
   - Monitor metrics and logs for errors.
   - Rotate webhook secrets if applicable.

Rollback plan

- If critical issue found immediately after release, revert the tag and deploy previous container.
- For SDK issues, publish a patch release (v1.0.1) with fixes.

Owners

- Release lead: Backend team
- QA: Integration tests and SDK test owners

Timing

- Target date: set per project calendar. Prefer a weekday morning for monitoring availability.
